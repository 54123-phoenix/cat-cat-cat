# 升级路径与 Harness 监督手册

本文档定义“激情而稳健”的升级执行方式：功能可以往前冲，但每一步都要接受 harness 检查。目标不是让检查成为负担，而是让团队在加功能时知道底线在哪里。

## 北极星

项目升级的北极星是：

```text
识别可信、档案可靠、社区活跃、传播自然、猫协可用
```

所有新功能都应该服务这条主闭环：

```text
识别猫猫 -> 记录偶遇 -> 查看档案 -> 参与社区 -> 分享传播 -> 回流继续参与
```

## 阶段路径

### P0 稳住主闭环

先让项目从“能演示”变成“能稳定内测”。

必须稳定的路径：

- 登录/注册
- 拍照识猫
- 候选确认或未知猫提交
- 偶遇记录
- 猫猫详情
- 社区发帖与互动

这一阶段不追求炫技，重点是清晰的错误态和可恢复体验。

### P1 做出传播尖刀

在稳定基础上做最有传播价值的功能：

- 公开猫猫主页
- 偶遇分享页
- 猫猫故事卡片或分享海报

重点是让“每只猫”和“每次偶遇”都能成为可分享内容。

### P2 让用户回来

把一次性访问变成持续关系：

- 关注猫猫动态通知
- 每日任务
- 猫猫贡献体系
- 本周明星猫与贡献榜

重点是让用户感觉自己在守护某几只猫，而不是只在使用工具。

### P3 强化猫协真实工作流

让项目服务真实猫协管理：

- 新猫线索审核增强
- 猫档案变更记录
- 健康记录可信来源
- 喂食点状态更新
- 举报审核工作台
- 管理端数据看板

重点是可信、可追踪、可管理。

### P4 做一个有记忆点的亮点

推荐亮点：

```text
校园猫路线推荐
```

基于最近偶遇、时间段和地点，为用户推荐一条“今天去哪遇见猫”的路线。这个功能展示效果强，但必须排在 P0-P2 之后。

## Harness 使用方式

快速检查：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File harness/run.ps1
```

完整检查：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File harness/run.ps1 -Mode full
```

继续跑完所有检查，即使中途失败：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File harness/run.ps1 -ContinueOnFailure
```

报告输出：

```text
harness/reports/latest.md
harness/reports/latest.json
```

## 分数规则

| 分数 | 结论 | 动作 |
| --- | --- | --- |
| 90+ | pass | 可以合入或发布候选 |
| 80-89 | internal-trial | 可以内测，但必须记录风险 |
| 70-79 | demo-only | 只能演示，不能上线 |
| <70 | stop-feature-work | 停止加功能，回到稳定性修复 |

## 功能完成定义

任何新功能不能只说“写完了”，必须满足：

- 代码完成
- 错误态可用
- 移动端主视图可用
- 主闭环不破坏
- quick harness 通过
- 阶段验收前 full harness 通过
- README 或相关文档更新
- 接受标准可逐条验证

## 任务写法

所有升级任务都写入 `harness/roadmap.yml`。推荐格式：

```yaml
- id: P1-01
  title: "公开猫猫主页"
  type: feature
  passion_score: 5
  risk_score: 2
  acceptance:
    - "免登录访问公开猫主页。"
    - "页面包含头像、名字、常出没地点、精选照片、最近偶遇。"
    - "登录用户可从公开页进入关注、打卡或查看完整档案。"
```

`passion_score` 代表展示价值、传播价值和用户兴奋度。
`risk_score` 代表工程复杂度、数据风险和维护成本。

高激情、高风险的任务必须拆小；低激情、高风险的任务应延后或取消。

## 推荐第一轮执行

第一轮建议只做这些：

1. P0 识别状态与降级体验统一
2. P0 上传安全与边界校验
3. P1 公开猫猫主页
4. P1 偶遇分享页
5. P2 关注猫猫动态通知

这组功能既能稳住主路径，也能制造传播和回流。
